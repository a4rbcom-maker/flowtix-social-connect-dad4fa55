## نتيجة المسح: مفيش تعبيرات تستدعي التقسيم

عملت مسح شامل لكل التعبيرات `${{ ... }}` داخل `run` و `if` و كل الـ workflow:

| المقياس | النتيجة |
|---|---|
| أطول تعبير `${{ }}` فردي | **54 حرف** |
| عدد التعبيرات > 100 حرف | **0** |
| عدد التعبيرات > 1000 حرف | **0** |
| الحد الذي يفرضه GitHub | **21000 حرف** |
| الهامش المتبقي | **99.7%** |

### توضيح مهم حول حد 21000

الحد ده بيطبّق على **التعبير الواحد** `${{ ... }}` (interpolation expression)، مش على طول الـ `run:` block أو السكربت ككل. السكربتات الطويلة في `run:` (زي اللي في L44 بحجم 6.3KB) **مش متأثرة بالحد ده** — هي شِل عادي.

والمشكلة اللي حصلت قبل كده مع `Exceeded max expression length 21000` كانت في تعبير واحد بيلف هيردوك ضخم — وده اتحل خلاص بنقل المنطق لـ `scripts/ci/install-restart.sh`.

### اقتراح: مفيش حاجة نعملها

أي تقسيم إضافي دلوقتي هيكون **تحسين بدون داعي** ويزود الـ overhead بدون فايدة. أكبر تعبير عندنا (54 حرف) أقل من الحد بـ ~388 ضعف.

### لو حابب احتياط استباقي

ممكن نضيف فحص CI خفيف (job منفصل) يكسر الـ build لو أي تعبير عدّى مثلاً 5000 حرف — guardrail يحمي من أي regression مستقبلي. لو موافق أضيفه، أو نسيب الموضوع كده ونقفل التذكرة.
