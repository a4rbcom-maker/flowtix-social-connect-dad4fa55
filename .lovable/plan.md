سأصلح السبب الظاهر جذريًا: الرسالة الحالية ليست خطأ تشغيل جديد، بل حالة قديمة محفوظة في قاعدة البيانات من الاختبار السابق الذي كان يحاول فتح `/me` من السيرفر. الخطة:

1. تحديث منطق تحميل حسابات بوت فيسبوك
   - عند قراءة الحسابات، إذا وجدنا `last_error` القديم: `صفحة /me أعادت محتوى تسجيل دخول`، نعيد تصنيفه تلقائيًا كحالة حديثة مفهومة بدل عرضه كفشل دائم.
   - للحسابات التي كانت محفوظة بهذه الرسالة القديمة، نجعل الحالة `untested` أو `active` بناءً على فحص بنية الكوكيز الحالي، بدل تركها `invalid` بدون سبب حقيقي.

2. توحيد فحص الكوكيز
   - استخدام نفس منطق `precheckBotAccount` داخل `testBotAccount` حتى لا توجد نتيجتان متعارضتان.
   - إذا الكوكيز الأساسية موجودة وصحيحة، تكون النتيجة: صالحة شكليًا وتنتظر VPS Worker للفحص الحقيقي.
   - إذا ناقصة أو منتهية، تظهر رسالة دقيقة باسم الكوكيز الناقصة/المنتهية.

3. تنظيف عرض الرسائل في صفحة بوت فيسبوك
   - عدم عرض رسالة `/me` القديمة نهائيًا في الجدول.
   - إضافة زر/مسار واضح: “تحديث الحالة” بدل “إعادة المحاولة” للحالات القديمة، حتى يفهم العميل أن النظام يعيد تقييم الحساب بالمنطق الجديد.

4. ترحيل بيانات آمن بدون تعطيل العملاء
   - إنشاء migration صغيرة تنظف فقط السجلات التي تحمل رسالة الخطأ القديمة، بدون حذف حسابات أو كوكيز أو مهام.
   - لن تتأثر الحسابات الشغالة أو المهام الحالية.

5. التحقق بعد التنفيذ
   - فتح `/dashboard/facebook/bot` والتأكد أن رسالة `/me` اختفت.
   - اختبار زر الفحص على الحساب الحالي والتأكد أن النتيجة الجديدة تظهر بدل الخطأ القديم.
   - مراجعة الشبكة/الكونسول للتأكد أنه لا يوجد crash أو فشل RPC.
