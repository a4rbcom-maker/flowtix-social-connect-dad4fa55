## التشخيص المختصر

المشكلة الحالية ليست “رسالة خطأ ناقصة” فقط. المسار نفسه أصبح هشًّا بسبب رقع متراكمة:

- الواجهة تستدعي دوال TanStack server functions عبر غلاف مخصص `useFacebookApi.call` بدل `useServerFn`، وهذا قد يكسر تكامل TanStack الطبيعي ويجعل شكل الرد/الأخطاء غير مضمون.
- `openPrecheck` يعتمد على “البحث داخل أي wrapper” بدل عقد واضح للرد، لذلك ظهر نص مثل: `استجابة غير متوقعة من الخادم. {}`.
- `precheckBotAccount` قد يرمي أخطاء خام من القراءة/فك التشفير/قاعدة البيانات بدل إرجاع DTO ثابت ومفهوم للواجهة.
- الواجهة تعرض الخطأ التقني داخل الصندوق الأحمر، بدل تحويل كل حالة إلى رسالة عملية للمستخدم.

## الخطة التنفيذية

1. **تثبيت عقد موحد لاختبار حساب البوت**
   - إنشاء أنواع واضحة داخل `src/lib/fb-bot.functions.ts` لنتيجة الفحص ونتيجة الاختبار.
   - جعل `precheckBotAccount` لا يرمي أخطاء قابلة للتوقع في مسار المستخدم؛ بل يرجع دائمًا شكلًا ثابتًا:
     - `ok`
     - `canContinue`
     - `severity`
     - `message`
     - `present/missing/invalid`
     - `expiresAt/expiresInDays/expired`
     - `debugCode` آمن بدون أسرار

2. **تحويل أخطاء السيرفر إلى رسائل عربية مفيدة**
   - معالجة حالات:
     - الحساب غير موجود
     - لا توجد جلسة/صلاحية
     - فشل فك تشفير الكوكيز
     - payload قديم أو غير مفهوم
     - JSON كوكيز ناقص
     - كوكيز منتهية
   - بدون عرض `{}` أو رسائل تقنية فارغة للمستخدم.

3. **إزالة الاعتماد الهش في `openPrecheck`**
   - استبدال الـ queue/wrapper walker بدالة normalization صغيرة ومحددة.
   - إن لم يصل رد صالح، تظهر رسالة ثابتة: “تعذّر قراءة نتيجة الفحص، أعد المحاولة أو حدّث الصفحة.” مع تسجيل التفاصيل للكونسول فقط.
   - إضافة ضمان UI: لا يتم عرض صندوق أحمر بدون نص أبدًا.

4. **إصلاح طريقة استدعاء دوال السيرفر في صفحة البوت**
   - استخدام `useServerFn` لدوال البوت الحساسة داخل `dashboard.facebook.bot.tsx` بدل تمرير الدالة الخام داخل `useFacebookApi.call`.
   - إبقاء `useFacebookApi` فقط لمسارات Facebook Graph العامة إذا لزم، وليس كطبقة عامة لكل server function.
   - هذا يعيد التكامل الطبيعي مع TanStack ويقلل مشاكل الـ wrappers والـ headers.

5. **تنظيف واجهة الاختبار**
   - تغيير زر “متابعة الاختبار” ليعتمد على `canContinue` بدل `ok` فقط، بحيث يمكن السماح بالاختبار في الحالات التحذيرية ومنعه في الحالات القاتلة.
   - عرض تفاصيل missing/invalid بطريقة منظمة بدل رسالة عامة.
   - إزالة اللوجات المؤقتة الزائدة أو جعلها مفيدة ومحدودة.

6. **اختبار مركز قبل التسليم**
   - اختبار السيناريوهات التالية من الكود بدون لمس أسرار المستخدم:
     - رد كوكيز ناقصة
     - رد كوكيز سليمة
     - رد كوكيز منتهية
     - رد غير متوقع
     - خطأ مصادقة
   - فحص أن الصندوق الأحمر لا يظهر فارغًا ولا يعرض `{}`.

## الملفات المتوقع تعديلها

- `src/lib/fb-bot.functions.ts`
- `src/routes/dashboard.facebook.bot.tsx`
- غالبًا `src/features/facebook/api.ts` بشكل محدود إذا احتجنا فصل غلاف Facebook Graph عن server functions

## النتيجة المتوقعة

بعد التنفيذ، مسار “اختبار الحساب” سيكون مبنيًا على عقد واضح بين السيرفر والواجهة، وليس ترقيعات على شكل الرد؛ وبالتالي أي فشل سيظهر برسالة مفهومة، ولن تظهر أخطاء فارغة أو `{}` مرة أخرى.
