المشكلة الظاهرة من الصورة أن خطوة النشر وصلت للسيرفر، فحصت الملفات، اكتشفت اختلاف/خلل في bundle، فعملت استرجاع تلقائي لآخر نسخة سليمة، ثم أفشلت الـ workflow بـ exit code 1. لذلك الموقع القديم ما زال يعمل، لكن النسخة الجديدة لم تُفعّل.

الخطة:

1. تعديل سكريبت النشر `scripts/ci/install-restart.sh`
   - بعد استرجاع snapshot بسبب فشل سلامة الملفات، لن يكتفي السكريبت بترك PM2 يعمل على الذاكرة القديمة.
   - سيعيد تشغيل/تحميل PM2 على النسخة المسترجعة نفسها حتى يصبح القرص والعملية الحالية متطابقين.
   - هذا يمنع بقاء السيرفر في حالة مربكة: ملفات على القرص من نسخة، وPM2 يشغّل نسخة أخرى.

2. جعل رسائل GitHub Actions أوضح
   - بدل رسالة عامة مثل `Process completed with exit code 1` فقط، سنضيف سبب مفهوم: هل الفشل بسبب checksum، file-list drift، أو rollback.
   - نوضح أن rollback نجح وأن الموقع لم ينقطع.

3. تقليل احتمالات فشل file-list drift الكاذب
   - مراجعة قائمة الملفات المستثناة من الفحص لتتطابق مع الملفات التي يستثنيها rsync.
   - إن كانت هناك ملفات محلية طبيعية على السيرفر تسبب اختلاف العدّ، سنضيفها للاستثناء بدل أن تكسر النشر.

4. الحفاظ على الأمان الحالي
   - لن نلغي فحص سلامة الـ bundle بالكامل، لأنه يحمي الموقع من تشغيل نشر ناقص أو تالف.
   - التعديل سيكون في طريقة التعافي والتشخيص، وليس تعطيل الحماية.

5. التحقق بعد التنفيذ
   - تشغيل فحوصات السكريبت/الـ workflow بشكل موضعي قدر الإمكان بدون تنفيذ deploy حقيقي.
   - بعدها تحتاج تعمل إعادة تشغيل للـ GitHub Action أو push جديد ليتم تطبيق نسخة النشر المعدلة على سيرفرك.
