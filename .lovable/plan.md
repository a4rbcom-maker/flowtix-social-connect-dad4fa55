المشكلة الآن ليست في شكل رسالة الواجهة فقط؛ الطلب نفسه يرجع HTTP 500 برسالة `HTTPError` عند استدعاء `precheckBotAccount`، وبعدها الواجهة تحاول عرض رد غير صالح فيظهر `{}`. الحل النهائي يجب أن يمنع الاثنين: فشل السيرفر غير المفهوم، ورسالة واجهة مبهمة.

## الخطة

1. **إزالة سبب 500 من مسار الفحص**
   - تعديل `precheckBotAccount` بحيث لا يترك أي خطأ يهرب من السيرفر، بما في ذلك أخطاء المصادقة/قراءة الحساب/فك التشفير/شكل البيانات.
   - أي فشل قابل للعرض يرجع DTO واضح بدل throw:
     - `ok`
     - `canContinue`
     - `severity`
     - `message`
     - `debugCode`
     - تفاصيل الكوكيز الناقصة/غير الصالحة.

2. **توحيد استدعاءات Server Functions في صفحة البوت**
   - استبدال استخدام wrapper `useFacebookApi.call` مع دوال البوت الحساسة باستدعاءات `useServerFn` مباشرة.
   - السبب: المشروع لديه `attachSupabaseAuth` في `src/start.ts`، وبالتالي `useServerFn` هو المسار الصحيح لإرسال جلسة المستخدم. الخلط الحالي بين wrapper يدوي وmiddleware هو مصدر أخطاء envelope/headers.
   - يظل `useFacebookApi` فقط لاستخدامات Facebook Graph إن وجدت، وليس CRUD/Facebook Bot server functions.

3. **تطبيع نتائج Server Functions مرة واحدة فقط**
   - إنشاء helper صغير داخل صفحة البوت لقراءة نتيجة السيرفر بأمان، لكن بدون “البحث العشوائي” في كل المفاتيح.
   - إن وصل رد غير معروف، تعرض الواجهة رسالة واضحة: “فشل الفحص من الخادم، أعد تسجيل الدخول أو حدّث الصفحة” مع `debugCode` ثابت، وليس `{}`.

4. **تعديل نافذة الفحص لتعرض تشخيص مفهوم دائمًا**
   - لا يتم عرض JSON raw للمستخدم.
   - كل الحالات تظهر نص عربي مباشر:
     - جلسة التطبيق منتهية
     - الحساب غير موجود
     - الكوكيز غير قابلة لفك التشفير
     - الكوكيز فارغة
     - كوكيز ناقصة
     - كوكيز منتهية
     - رد غير متوقع من الخادم
   - زر “متابعة الاختبار” يعتمد فقط على `canContinue`.

5. **إصلاح زر الاختبار نفسه بنفس المعمارية**
   - `testBotAccount` يُستدعى عبر `useServerFn` مباشرة.
   - نتيجة الاختبار تُطبّع إلى Account واضح قبل تحديث الجدول.
   - أي فشل يتحول إلى Toast عربي مفهوم بدل `unknown` أو `{}`.

6. **إضافة logging آمن ومؤقت للتشخيص**
   - تسجيل `debugCode` وشكل الرد فقط بدون كوكيز أو بيانات حساسة.
   - الهدف أن أي خطأ لاحق يظهر سببه خلال دقيقة بدل ساعات.

7. **التحقق بعد التنفيذ**
   - أتحقق من مسار `precheckBotAccount` بعد التعديل وأنه لا يرجع 500.
   - أتحقق من أن نافذة الفحص لا تعرض `{}` أبدًا.
   - أتحقق من سيناريو عدم وجود جلسة/رد غير متوقع/كوكيز ناقصة.
