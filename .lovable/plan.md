## التشخيص الحالي

من فحص الدومين مباشرة، الموقع يرجّع HTML بنجاح، لكن أغلب ملفات `/assets/*.js` و`/assets/*.css` المطلوبة ترجع 404. هذا يفسّر أن الموقع “بايظ”: صفحة HTML موجودة، لكن ملفات الواجهة التي يشير لها الـ HTML غير موجودة في مكانها الصحيح.

السبب الأكثر احتمالًا من إعداد النشر الحالي:

- الـ workflow يرفع فقط محتويات `.output/` إلى:
  `/www/wwwroot/flowtixtools.com/.output/`
- بينما الـ HTML المنشور يطلب الملفات من:
  `https://www.flowtixtools.com/assets/...`
- إذن Nginx/السيرفر يبحث عن assets في:
  `/www/wwwroot/flowtixtools.com/assets/`
  لكنها غالبًا موجودة داخل `.output/public/assets/` أو لم تُنسخ للمسار العام.
- هذا لا يعني أن GitHub لا يسحب؛ GitHub Actions ناجح، لكن الـ deployment غير مكتمل أو Nginx لا يوجّه static assets للمكان الصحيح.

## الهدف

إصلاح نشر هذا المشروع فقط، بدون لمس إعدادات عامة على الخادم أو التأثير على أي مواقع أخرى.

## الخطة المقترحة

1. **تثبيت شكل الـ build الصحيح للمشروع**
   - نترك إعداد `node-server` لأن السيرفر عندك عليه Node وPM2.
   - نضبط workflow ليعرف بوضوح أين توجد ملفات التشغيل وملفات static assets بعد البناء.

2. **تعديل GitHub Actions ليعمل نشر آمن ومكتمل**
   - رفع `.output/` كاملًا لتشغيل SSR/Node.
   - نسخ ملفات `.output/public/` أيضًا إلى جذر الموقع `/www/wwwroot/flowtixtools.com/` حتى تعمل روابط `/assets/...` مباشرة.
   - إضافة تحقق صريح بعد الرفع على السيرفر:
     - هل ملف server entry موجود؟
     - هل مجلد `assets` موجود؟
     - هل يوجد CSS/JS فعليًا؟
   - لو أي شيء ناقص، الـ Action يفشل بدل ما يظهر أخضر كاذب.

3. **جعل Restart الخاص بـ PM2 لا يلمس غير هذا المشروع**
   - استخدام اسم تطبيق واضح وفريد مثل `flowtixtools-web` بدل اسم عام قد يتعارض مع مشاريع أخرى.
   - restart/start لهذا الاسم فقط.
   - عدم استخدام أي أوامر توقف PM2 بالكامل أو reload عام.

4. **تأمين المسار حتى لا يحدث حذف خاطئ**
   - استخدام `rsync --delete` فقط داخل مسار المشروع المحدد.
   - عدم حذف أو لمس `/www/wwwroot/` بالكامل.
   - عدم تعديل إعدادات Nginx العامة من داخل workflow.

5. **إضافة تشخيص مفيد داخل GitHub Actions**
   - طباعة محتويات `.output` المهمة بعد البناء.
   - طباعة الملفات الموجودة على السيرفر بعد الرفع لهذا المشروع فقط.
   - طباعة حالة PM2 للتطبيق المحدد فقط.

## المطلوب منك بعد التنفيذ

بعد تعديل الملفات، ستعمل push/Sync إلى GitHub، ثم تفتح GitHub Actions وتشوف run جديد. لو نجح، الموقع يجب أن يتوقف عن طلب assets مفقودة.

لو ظل الموقع لا يعمل بعد ذلك، غالبًا سيكون السبب من Nginx vhost نفسه، وساعتها نحتاج فقط نراجع بلوك `server {}` الخاص بـ `flowtixtools.com` بدون لمس أي مواقع أخرى.
